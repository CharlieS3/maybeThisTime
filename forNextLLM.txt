## Database schema (current)
```sql
profiles (
  id UUID PK DEFAULT gen_random_uuid(),
  first_name    TEXT NOT NULL,
  last_name     TEXT NOT NULL,
  email         TEXT NOT NULL UNIQUE,
  username      TEXT NOT NULL UNIQUE,   -- login identifier
  phone         TEXT,                   -- nullable: optional
  password_hash TEXT NOT NULL           -- argon2id hash; NEVER the password itself
)
breweries (
  id   UUID PK DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  code TEXT NOT NULL UNIQUE             -- server-generated 6-char code
)
memberships (
  profile_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  brewery_id UUID NOT NULL REFERENCES breweries(id) ON DELETE CASCADE,
  role TEXT NOT NULL CHECK (role IN ('OWNER', 'ADMIN', 'BREWER', 'STAFF')),
  PRIMARY KEY (profile_id, brewery_id)
)
sessions (                              -- NEW (migration 007, applied)
  id UUID PK DEFAULT gen_random_uuid(), -- the session ID itself; PK = fast lookup
  profile_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),  -- debugging/stats only
  expires_at TIMESTAMPTZ NOT NULL       -- set by login code, not a table default
)
schema_migrations (filename TEXT PK, applied_at TIMESTAMPTZ NOT NULL DEFAULT now())
```
- Never edit applied migration files — add new numbered ones. Adding NOT NULL columns to a table with rows fails; in dev we delete the test rows first (done twice now).
- Roles stored UPPERCASE; `@IsIn` rejects lowercase at the API, service also does `.toUpperCase()`.
- Composite PK on memberships: can't join same brewery twice; CAN belong to multiple breweries with different roles.
- Editor highlights `sessions` blue (SQL-keyword-ish highlighting); harmless, not reserved in Postgres.

## Business rules
- **A brewery can exist on its own; a profile cannot.** `ProfilesService.create` inserts profile + membership in one DB transaction.
- **Brewery codes are server-generated, never client-supplied.** 6 chars, 32-char alphabet (no 0/O/1/I), `crypto.randomInt`, retry up to 5x on 23505 collision. Code = identifier; the password carries the security load.
- **Passwords:** argon2id (over bcrypt — OWASP first pick, memory-hard, no 72-byte truncation). Min length 12, max 128, NO composition rules (NIST/OWASP: length beats forced symbols; later add a common-password blocklist instead). Hash computed in service BEFORE/OUTSIDE the transaction. `password_hash` appears in NO RETURNING and NO SELECT — except the single lookup inside `AuthService.login` (needed there as the comparison target for `argon2.verify(hash, attempt)`); it never leaves the method.
- **Future sensitive data**: passwords = hash (one-way); usable secrets like Square API tokens = encryption at rest or secret manager, NEVER hash; business data like inventory = protected by authz/tenancy, not crypto.

## Auth — sessions (THIS SESSION'S WORK)
Concepts covered with me (I'm comfortable with these now):
- HTTP is stateless → session = server-generated random ID stored in a `sessions` table + handed to browser as a cookie ("coat-check ticket" analogy).
- Sessions intentionally SURVIVE browser close; two separate lifetimes: cookie lifetime (browser) vs `expires_at` (server row). Row is removed by logout (explicit DELETE) or treated dead by expiry check (`expires_at > now()` on lookup); old-row sweeping is cosmetic cleanup later.
- httpOnly cookie = page JS can't read it (XSS), the core reason cookies were chosen over JWT-in-localStorage.

## Validation layer
- `class-validator` + `class-transformer`; global `ValidationPipe({ whitelist: true, forbidNonWhitelisted: true })` in main.ts.
- Mental model: pipe = "is the request well-formed?" (in isolation); DB constraints + exception filter = "does it make sense against current data?".
- Decorators in use: `@IsString`, `@Length`, `@IsUUID`, `@IsIn`, `@IsEmail`, `@IsOptional`, `@IsNotEmpty`.
- Gotchas learned: `@IsOptional` skips only null/undefined, NOT "" (frontend sends `phone: phone || undefined`); `@IsString` redundant under `@IsIn(strings)`/`@IsEmail`; `isUUID` (function) vs `@IsUUID` (decorator); every endpoint validates independently.

## DB error mapping
- `src/database/database-exception.filter.ts`: `@Catch(DatabaseError)` (pg's error class) — DB errors only; everything else keeps default handling. Registered with `app.useGlobalFilters(...)`; order vs pipes irrelevant (pipes inbound, filters outbound).
- 23505 unique_violation → 409 "Resource already exists"; 23503 foreign_key_violation → 400 "Referenced resource does not exist"; default → clean 500.

## Conventions
- DB = snake_case, API/frontend = camelCase; translate via SQL aliases `AS "camelCase"` (double quotes). Single-word columns need no alias.
- Explicit column lists, no SELECT \* (this is the mechanism keeping password_hash from leaking).
- Parameterized queries always; SQL-computed expressions (e.g. `now() + interval '7 days'`) written directly in the SQL string.
- Optional values: `dto.phone ?? null`. Clients never supply `id` or `code`. DTO fields: `!` required, `?` optional.
- Controller = HTTP only; Service = logic + SQL; only DatabaseService touches the pool. In transactions: `client.query`, never `this.db.query`.
- Frontend form fetch pattern: try/catch + `if (!res.ok)` + err.message array-or-string; error branch keeps inputs, success branch clears ALL field states.
- Server-component list pages: `let data: T[];` + try/catch + `if (!res.ok) throw` + friendly fallback return in catch.
- Controlled inputs: `value={X}` and `onChange → setX` must reference the SAME state.
- Confirm-password is frontend-only; checked at top of handleSubmit; never in the DTO.
- `type="password"` on password inputs. Login is POST, never GET.
- Types: `phone: string | null`. Key order never matters in objects/types; only positional `$1, $2` arrays care.
- New modules: `npx nest g module|controller|service <name>` from `apps/api`; auth module imports DatabaseModule.
