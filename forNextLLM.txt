## Database schema (current)
```sql
profiles (
  id UUID PK DEFAULT gen_random_uuid(),
  first_name TEXT NOT NULL,
  last_name  TEXT NOT NULL,
  email      TEXT NOT NULL UNIQUE,
  username   TEXT NOT NULL UNIQUE,   -- future login identifier
  phone      TEXT                    -- nullable: optional
)
breweries (
  id   UUID PK DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  code TEXT NOT NULL UNIQUE          -- server-generated 6-char join/login code
)
memberships (
  profile_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  brewery_id UUID NOT NULL REFERENCES breweries(id) ON DELETE CASCADE,
  role TEXT NOT NULL CHECK (role IN ('OWNER', 'ADMIN', 'BREWER', 'STAFF')),
  PRIMARY KEY (profile_id, brewery_id)
)
schema_migrations (filename TEXT PK, applied_at TIMESTAMPTZ NOT NULL DEFAULT now())
```
- Migration runner skips files already recorded in `schema_migrations`; never edit applied migration files — add new numbered ones.
- Lesson learned this session: adding NOT NULL columns to a table with rows fails — we chose to delete dev rows first rather than use defaults or nullable columns.
- Roles are stored UPPERCASE; CHECK is case-sensitive. Service normalizes with `dto.role.toUpperCase()`, and `@IsIn` now also rejects lowercase before it reaches the service.
- Composite PK on memberships = same person can't join the same brewery twice, but CAN belong to multiple breweries (possibly with different roles).

## Business rules we adopted
- **A brewery can exist on its own; a profile cannot.** Every profile must be created with a brewery + role. Enforced in the service layer: `ProfilesService.create` inserts profile + membership in **one DB transaction**.
- **Brewery codes are server-generated, never client-supplied** (same principle as IDs: unique/unguessable values are not the client's job). 6 chars from a 32-char alphabet (no 0/O/1/I — ambiguity), `crypto.randomInt` (not Math.random — codes are credential-adjacent). 32^6 ≈ 1B combinations; code is an identifier, the password (future) carries the security load.
- **Normalization:** names live only in profiles/breweries; memberships stores only ids + role.

## Validation layer (NEW this session)
- `class-validator` + `class-transformer` installed in `apps/api`.
- Global pipe in `main.ts`: `new ValidationPipe({ whitelist: true, forbidNonWhitelisted: true })` — strips undeclared properties and 400s if extras are sent.
- Mental model we settled: **pipe = "is this request well-formed?" (checks body in isolation); DB constraints + exception filter = "does it make sense given current data?"** A request can pass the first and legitimately fail the second.
- Decorators in use: `@IsString`, `@Length(min, max)`, `@IsUUID`, `@IsIn([...])`, `@IsEmail`, `@IsOptional`.
- Gotchas learned:
  - `@IsOptional` only skips `null`/`undefined`, NOT `""` — frontend must send `phone: phone || undefined`.
  - `@IsString` is redundant under `@IsIn(strings)` or `@IsEmail` (kept off).
  - `isUUID` (lowercase) is a function, `@IsUUID` is the decorator — casing matters.
  - Each endpoint validates independently; never assume "another DTO already checked this."

## DB error mapping (NEW this session)
- `src/database/database-exception.filter.ts`: `@Catch(DatabaseError)` (the class `pg` throws) — only DB errors enter it; ValidationPipe 400s etc. keep default handling.
- Registered globally: `app.useGlobalFilters(new DatabaseExceptionFilter())`. Order vs pipes doesn't matter (pipes = inbound, filters = outbound).
- Mapping: `23505` unique_violation → 409 "Resource already exists"; `23503` foreign_key_violation → 400 "Referenced resource does not exist"; default → clean 500 (no leaked details).
- `BreweriesService.create` retries code generation up to 5x on 23505 (collision), rethrows anything else for the filter.

## Conventions we settled on
- **DB = snake_case, API/frontend = camelCase.** Translation via SQL aliases: `last_name AS "lastName"` (double quotes required). Single-word columns (email, username, phone, code) need no alias.
- **Explicit column lists, no SELECT \*.** Cost acknowledged: every new column means updating SELECTs/INSERTs — worth it.
- Parameterized queries always (`$1`, values array). Optional values passed as `dto.phone ?? null` (explicit over implicit).
- Clients never supply `id` or `code` — server/DB generates. DTOs exclude them.
- DTO properties use `!` for required, `?` for optional (e.g. `phone?: string;` — `!` on an optional field is a lie to TypeScript).
- Controller = HTTP only; Service = logic + SQL; only DatabaseService touches the pool. Inside a transaction callback, always `client.query`, never `this.db.query`.
- Frontend fetch pattern (forms): try/catch (network failure) + `if (!res.ok)` (fetch does NOT throw on 4xx/5xx) + handle `err.message` as array (ValidationPipe) or string (filter). Error branch keeps inputs; success branch clears ALL field states.
- Frontend fetch pattern (server-component list pages): `let data: T[];` + try/catch + `if (!res.ok) throw` + friendly fallback page in catch.
- Types: `phone: string | null` (SQL NULL → JSON null). Property order never matters in objects/types; only positional things like `VALUES ($1, $2)` care about order.
- Next.js: routes mirror screens, not actions. Server components for read pages; `"use client"` only where interactive. Hooks only at component top level (never inside handlers).

## Working endpoints / pages (all verified)
- GET  /profiles    → `[{ id, firstName, lastName, username, email, phone }]`
- POST /profiles    → body `{ firstName, lastName, username, email, phone?, breweryId, role }`; transaction creates profile + membership
- GET  /breweries   → `[{ id, name, code }]`
- POST /breweries   → body `{ name }`; returns row incl. generated `code`
- GET  /memberships → JOINs → `[{ profileId, firstName, breweryId, breweryName, role }]`
- POST /memberships → body `{ profileId, breweryId, role }`
- Pages: /profiles, /profiles/new (full form), /breweries, /breweries/new (shows code on success), /memberships — all with error handling
