## AUTH — STATUS: backend COMPLETE and verified; frontend ~half done

### Backend (all done, all curl-verified)
1. **Login sets httpOnly cookie** — `AuthController.login` uses `@Res({ passthrough: true })`, `res.cookie('session_id', sessionId, { httpOnly: true, sameSite: 'lax', maxAge: 7*24*60*60*1000 /* secure: true in prod */ })`, returns user WITHOUT sessionId in body. The old ts(2339) blocker fixed itself once `import type { Response } from 'express'` was in place + TS server caught up (`@types/express@5.0.6` present).
2. **cookie-parser wired** in main.ts: `import cookieParser from 'cookie-parser'` (DEFAULT import — `import * as` fails with ts(2349) under our esModuleInterop config) + `app.use(cookieParser())` → `req.cookies` exists everywhere.
3. **CORS**: `app.enableCors({ origin: 'http://localhost:3001', credentials: true })` — credentials flag is required or browsers discard cookies cross-origin. Pairs with `credentials: "include"` on frontend fetches (both sides must opt in).
4. **Logout**: `AuthService.logout(sessionId)` = `DELETE FROM sessions WHERE id = $1`. `AuthController.logout` = `@Post('logout')`, reads `req.cookies.session_id` via `@Req`, calls service only if present, always `res.clearCookie('session_id')`, always returns `{ success: true }` (nothing useful to error about). Verified: response shows `Set-Cookie: session_id=; Expires=Thu, 01 Jan 1970` (deletion = past expiry date).
5. **SessionGuard** (`src/auth/session.guard.ts`):
   - `@Injectable`, `implements CanActivate`, injects DatabaseService (works like in services).
   - `const req = context.switchToHttp().getRequest();`
   - `if (!sessionId || !isUUID(sessionId)) throw new UnauthorizedException();` — `isUUID` is the FUNCTION from class-validator (lowercase i; capital `@IsUUID` is the DTO decorator).
   - One query: `SELECT p.id, p.username, p.first_name AS "firstName", p.last_name AS "lastName" FROM sessions s JOIN profiles p ON p.id = s.profile_id WHERE s.id = $1 AND s.expires_at > now()`
   - No row → same `UnauthorizedException`. Row → `req.user = result.rows[0]; return true;`
6. **All five endpoints guarded** with `@UseGuards(SessionGuard)` per-route: GET/POST profiles, GET/POST breweries, GET /memberships, POST /memberships. (Login/logout deliberately unguarded — login CREATES the session; guard it and nobody can ever log in.)

## Concepts I'm now comfortable with (this session's additions)
- Nest request pipeline ORDER is fixed: middleware → guards → pipes → controller. That order is why the guard can rely on req.cookies (cookie-parser already ran).
- Guard = bouncer answering ONE yes/no question ("may this request proceed?"); throws 401 or returns true; on throw the controller never runs. Attaches `req.user` = name tag for later tenancy/role checks. `@UseGuards` is per-route/per-controller opt-in; same guard reusable everywhere; global-guard-with-@Public() pattern exists for later; guards stack (`@UseGuards(SessionGuard, RolesGuard)`).
- Generics: `transaction<T>(fn: (client) => Promise<T>): Promise<T>` = "whatever the callback returns, transaction returns" — one reusable function WITH precise types (vs hardcoding a type or `any`).
- `client` in transactions = the one pool connection where BEGIN ran; pool.query could grab a different connection outside the transaction.
- Cookie lifetimes recap: logout kills BOTH (row DELETE + clearCookie); browser-close kills neither; 7-day expiry kills both independently. New login = new row; other devices' sessions unaffected.
- Defense layers on the cookie path: guard's format check (junk never reaches Postgres) → parameterized $1 (data, never SQL) → exception filter (clean 500 if something still escapes).
- A 500 isn't a crash — the server keeps serving; filter contained it. But wrong-status + log noise + info leak are why we fixed it to 401.
