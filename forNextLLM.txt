## Database schema (current)
```sql
profiles  (id UUID PK DEFAULT gen_random_uuid(), first_name TEXT NOT NULL)
breweries (id UUID PK DEFAULT gen_random_uuid(), name TEXT NOT NULL)
memberships (
  profile_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  brewery_id UUID NOT NULL REFERENCES breweries(id) ON DELETE CASCADE,
  role TEXT NOT NULL CHECK (role IN ('OWNER', 'ADMIN', 'BREWER', 'STAFF')),
  PRIMARY KEY (profile_id, brewery_id)
)
schema_migrations (filename TEXT PK, applied_at TIMESTAMPTZ NOT NULL DEFAULT now())
```
- Migration runner skips files already recorded in `schema_migrations`; never edit applied migration files — add new numbered ones (ALTER TABLE etc.).
- Roles are stored UPPERCASE; CHECK is case-sensitive. Services normalize with `dto.role.toUpperCase()` before INSERT.
- Composite PK on memberships = same person can't join the same brewery twice, but CAN belong to multiple breweries (possibly with different roles).

## Business rules we adopted
- **A brewery can exist on its own; a profile cannot.** Every profile must be created with a brewery + role. Enforced in the service layer (not the DB): `ProfilesService.create` inserts profile + membership in **one DB transaction** (all-or-nothing; bad breweryId rolls back the profile too). Old brewery-less profiles were deleted.
- **Normalization:** names live only in profiles/breweries; memberships stores only ids + role. Names are assembled via JOIN when needed (see GET /memberships).

## Conventions we settled on
- **DB = snake_case, API/frontend = camelCase.** Translation happens in service-layer SQL via aliases: `SELECT id, first_name AS "firstName" FROM profiles` (double quotes required).
- **Explicit column lists, no SELECT \*** (deliberate exposure of fields; security).
- Parameterized queries always (`$1`, values array).
- Clients never supply `id` — DB generates UUIDs. DTOs exclude id.
- DTO properties use `!` (e.g. `firstName!: string;`). Controllers pass the whole DTO to services (settled this session).
- Controller = HTTP only; Service = logic + SQL; only DatabaseService touches the pool.
- DatabaseService now has two methods: `query(sql, values)` for single queries, and `transaction(fn)` for all-or-nothing groups — checks out one PoolClient, BEGIN → fn(client) → COMMIT, ROLLBACK on error, release in finally. Inside a transaction callback, always use `client.query`, never `this.db.query`.
- Next.js: routes mirror screens, not actions (`/profiles`, `/profiles/new`; delete will be a button, not a page). Server components fetch for read pages; `"use client"` only where interactivity is needed. `/profiles/new` uses `useEffect(..., [])` to load the brewery dropdown once on mount.
- INSERTs use `RETURNING id, ... AS "camelCase"` to send the created row back.
- New modules generated with `npx nest g module|controller|service <name>` from `apps/api`.

## Working endpoints / pages (all verified)
- GET  /profiles    → `[{ id, firstName }]`
- POST /profiles    → body `{ firstName, breweryId, role }`; transaction creates profile + membership; returns `{ id, firstName, breweryId, role }`
- GET  /breweries   → `[{ id, name }]`
- POST /breweries   → body `{ name }`, returns created row
- GET  /memberships → JOINs profiles + breweries → `[{ profileId, firstName, breweryId, breweryName, role }]`
- POST /memberships → body `{ profileId, breweryId, role }` (adds an existing profile to another brewery)
- Pages: /profiles, /profiles/new (name + brewery dropdown + role dropdown, works end to end), /breweries, /breweries/new

## Known gaps / deliberately deferred
- **No auth/authz yet.** Plan: login (sessions or JWT), role checks using memberships.role, tenancy scoping in SQL (`WHERE brewery_id = $1` from the authenticated user, never from client input).
- No input validation (class-validator/ValidationPipe not set up). E.g. empty names accepted; bad role reaches the DB and 500s instead of a clean 400.
- No error handling anywhere: frontend fetches are happy-path only; API lets DB errors (FK violations, duplicate membership PK, CHECK failures) surface as 500s.
- No /memberships page in the frontend yet (data only visible via curl/DB tool).
- **seedProfile.ts now violates the "profile must have a brewery" rule** (inserts a bare profile) — needs updating or retiring.
- Default Nest spec files untouched (will need mock providers later).
- packages/database has no npm scripts (declined for now; must run tsx from `packages/database`).
- Seeds insert duplicates every run (accepted for now).
- pgcrypto extension line may be unnecessary if Postgres ≥ 13 (gen_random_uuid is built in).
